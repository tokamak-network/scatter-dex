## 1. 확정 신규 항목 (actionable)

### A-1 · [MEDIUM · 프라이버시] 공개 trade-offers 엔드포인트가 트레이더 식별자 노출
- **위치:** `zk-relayer/src/routes/relayer-stats.ts:68` → `core/db.ts:398` (`SELECT * FROM trade_offers`)
- **노출 컬럼:** `peer_relayer`, `maker_pub_key`, `taker_pub_key`, `maker_nonce`, `taker_nonce`, `reason`, `tx_hash`
- **불일치 근거:** 동일 데이터의 admin 라우트 `GET /api/admin/trade-offers`(`routes/admin.ts:816-827`)는 `adminAuth`로 게이트되어 있고, 주석에 "operator-private peer associations / 식별자 누출 방지"라고 명시. 그런데 공개 라우트(`/api/relayer/trade-offers`, `index.ts:467`, readLimiter only)가 같은 컬럼을 그대로 반환.
- **익스플로잇:** 비인증 클라이언트가 전체 교차-릴레이어 audit trail을 스크랩 → 각 정산 `tx_hash`를 maker/taker **EdDSA pubkey** 및 상대 relayer와 상관. 프라이버시 중심 DEX의 de-anonymization (자금 손실 없음).
- **검증:** ✅ 직접 확정
- **수정:** 공개 응답을 안전 subset(status/tx_hash/created_at/direction)으로 projection, pubkey/nonce/peer/reason 제거.

### A-2 · [MEDIUM · DoS] RemoteOrderStore 무제한 증가 (오더북 상한 미적용)
- **위치:** `zk-relayer/src/core/remote-orderbook.ts:39` (`add()` 상한 없음), `index.ts:42` (`MAX_ORDERBOOK_SIZE = 10_000` **정의만 되고 미사용** — 참조 0건)
- **조건:** `/api/p2p`는 rate limiter 없이 마운트, `verifyRelayerAuth`는 permissionless(자기 선언 주소로 서명만 맞으면 통과, allowlist 없음, `minBond` 기본 0).
- **익스플로잇:** 유니크 id·먼 만료의 OrderSummary를 `POST /api/p2p/orders`로 플러딩 → `byId`/`byRelayer`/정렬 pair 리스트가 무제한 증가(삽입 O(N) → 총 O(N²)) → 힙/CPU 고갈. `purgeExpired`는 만료 시에만 회수하나 공격자가 만료를 멀리 설정.
- **검증:** ✅ 직접 확정 (MAX_ORDERBOOK_SIZE 미사용 grep 확인)
- **수정:** `add()`에 `MAX_ORDERBOOK_SIZE` 실제 적용(LRU/거부), `/api/p2p`에 rate limit, 원격 만료 상한 검증.

### A-3 · [MEDIUM · DoS] shared-orderbook settlement 행 무제한 증가
- **위치:** `shared-orderbook/src/core/db.ts:745` (`insertSettlement`, 상한 없음), `db.ts:1056` (`getRelayerSettlementStats`가 전 행을 힙에 materialize), `routes/settlements.ts:51`
- **조건:** `POST /api/settlements`에 registry/bond 게이트 없음(아무 키페어나 `relayerAuth` 통과). authZ는 `submitter ∈ {makerRelayer, takerRelayer}`인데 둘 다 공격자 제출값. `takerRelayer`를 임의 피해자로 지정해 타인에게 행 귀속 가능. `txHash`만 유니크 제약, 행 상한/pruning 없음.
- **익스플로잇:** 가짜 행 누적 → 공개 `GET /api/relayers/:addr/stats`가 매 요청마다 해당 주소 전 행을 Node 힙에 적재 → 메모리/CPU 고갈. per-relayer limiter는 신원 회전으로 우회, per-IP limiter만 브레이크.
- **검증:** 에이전트 보고 (직접 재검증 미실시)
- **수정:** 쓰기를 registry 멤버십으로 게이트 + 행 상한/pruning + nullifier 포맷 검증.

### A-4 · [LOW-MEDIUM · 그리핑] P2P trade-offer proof 미검증 + rate limit 없음
- **위치:** `zk-relayer/src/routes/p2p.ts:172` → `authorize-cross-relayer-matcher.ts handleTradeOffer`
- **갭:** 로컬 제출 경로(`routes/authorize-orders.ts:370`)는 `config.verifyOrderProofs` 따라 Groth16 오프체인 검증하지만, `handleTradeOffer`는 taker proof를 검증하지 않고 token/price/expiry/fee만 보고 `submitAuthSettle` 호출 → 글로벌 `txMutex` + `eth_estimateGas` 소비.
- **익스플로잇:** permissionless peer가 공개된 maker id 참조 junk proof를 플러딩 → 정산이 mutex 뒤에 직렬화되고 RPC 쿼터 소모(브로드캐스트 전 estimateGas revert라 자금/가스 손실은 없음).
- **검증:** 에이전트 보고
- **수정:** mutex/estimate 전에 `verifyAuthorizeProof(takerOrder)` 호출 + `/api/p2p` rate limit.

### A-5 · [LOW-MEDIUM · 모니터링 오염] 검증불가 settlement 행 영구 미-pruning
- **위치:** `shared-orderbook/src/core/verifier.ts` (match된 행만 `verified=1` 플립), `routes/admin.ts:101` (`oldestUnverifiedBlock`)
- **익스플로잇:** 가짜 행(`blockNumber=0`) 1건 → `GET /api/admin/verify-stats`가 `hasUnverifiedRows:true`, `oldestUnverifiedBlock:0`을 영구 보고 → "미검증 백로그/oldest block" 알림이 영구 트리거되어 실제 검증 실패를 가림.
- **검증:** 에이전트 보고
- **수정:** "pending" vs "verify-impossible" 상태 구분 + 실패 행 격리/pruning.

### A-6 · [LOW · 그리핑] KYC submit 지갑 소유 증명 없음
- **위치:** `shared-orderbook/src/routes/kyc.ts:157` (`POST /api/kyc/submit`, public, writeLimiter only)
- **익스플로잇:** `wallet`이 서명 바인딩 없는 평문 필드. 공개 `GET /api/kyc/status?wallet=`로 피해자 pending 상태 확인 후 동일 wallet로 제출 → 피해자 email/video/idDoc 덮어쓰기 + 큐 위치 리셋. 또한 임의 wallet마다 디스크 폴더(최대 25MB×2) 생성 → 디스크 채우기.
- **검증:** 에이전트 보고

### A-7 · [LOW · 재평가] apps/pro 활성 배포 타겟인데 CSP 없음

- **검증:** 에이전트 보고 (기지 #34이나 리스크 전제 변경)

## 2. 사용자 제보 항목 (직접 검증 완료)

### B-1 · [MEDIUM · 컴플라이언스] KYC 승인이 운영 중 재검증되지 않음
- **위치:** 등록 시 KYC 체크 `RelayerRegistry.sol:205-211` ↔ 운영 판정 `_isOperational` `RelayerRegistry.sol:364-372` (identity만 재검증, KYC 미확인). 주석 `RelayerRegistry.sol:528-531`: "already-seated relayers are never re-checked."
- **설명:** `register()`는 `identityRegistry.isVerified` AND `kycApprovalRegistry.isApproved`를 체크. 그러나 `_isOperational`(정산 게이트)은 `identityRegistry.isVerified`만 연속 재검증하고 **KYC 승인은 보지 않음**. → KYC 승인이 철회됐지만 zk-X509 인증서가 유효한 relayer는 계속 정산 가능.
- **영향:** "승인 유지"가 요구되는 KYC 정책 환경에서 컴플라이언스 우회 상태. (기술적 자금탈취보다 정책/거버넌스 리스크)
- **검증:** ✅ 직접 확정
- **수정:** `_isOperational`에 운영 모드 기준 KYC 승인 재검증 포함(옵션이 아니라 정책 모드로). identity와 동일하게 fail-closed + try/catch로 게이팅.

### B-2 · [LOW · 설정의존] Bond fee-on-transfer/rebasing 시 회계 불일치
- **위치:** `RelayerRegistry.sol:316-328` (`_pullBond`이 delta 아닌 입력 `bondAmount` 반환·기록), 주석 invariant `RelayerRegistry.sol:305-313`
- **설명:** ERC20 path가 `safeTransferFrom(bondAmount)` 후 **요청값 그대로** 반환·저장(balance delta 미측정). 주석에 "no fee-on-transfer / rebasing bond tokens" invariant로만 의존하고 강제 검증 없음. `CommitmentPool.deposit`은 delta 검증을 하지만 bond path는 안 함.
- **영향:** 잘못된 bond token 설정 시 기록 bond > 실제 수령 → exit 시 초과 지급/고갈. (owner 설정 실수 의존, LOW)
- **검증:** ✅ 직접 확정 (in-code 문서화된 known invariant)
- **수정:** balance-before/after delta 기반 기록, 또는 표준 ERC20 allowlist 강제.

## 4. 견고 확인 (verified safe)

- **회로(circom):** 모든 public signal이 실제 산술 제약으로 witness에 바인딩(`x*x` anti-opt 외). nullifier 도메인 분리(tags 0/1/2/3), Merkle `leafIndex===ΣpathIndices·2^i`, `pathIndices∈{0,1}`, EdDSA 하드 enable, bit-width 정상. **신규 soundness 갭 없음.** (settle/batch는 별도 회로가 아니라 authorize 2개 결합/claim 루프)
- **핵심 정산:** `SettleVerifyLib` 교차검증(C1 토큰·C2 가격·C4 `totalLocked+fee≤상대 sellAmount`·fee·expiry), 모든 정산 경로 `spendEscrowNullifier` 단일화(PR#1074 정합), 전 엔트리포인트 `nonReentrant`+CEI, claimsRoot 등록 중복 차단, `isKnownRoot` 링버퍼, claim 회계(`totalClaimed+amount≤totalLocked`, uint128 cap, token/releaseTime/sanctions/identity 게이트), SafeERC20+fee-on-transfer 거부(deposit/settleWithDex balance-delta).
- **periphery/접근제어:** 전 impl `_disableInitializers`+`initializer`, UUPS 없음(transparent admin만), `Ownable2Step`+renounce 비활성, bond 토큰 per-relayer 스냅샷, `executeExit` CEI, FeeVault 잔액 invariant(`_assertBalanceBacked`), Treasury 보고자 게이트+attributed 플로어.
- **zk-relayer:** order/claim 경로 Groth16 오프체인 검증(fail-closed), `VERIFY_ORDER_PROOFS` 기본 ON, admin SIWE 게이트(미연결 시 fail-closed), 개인키 미로깅, better-sqlite3 prepared params(SQL injection 없음), 정산 SSRF `redirect:"error"`(PR#1082), txMutex 논스 직렬화.
- **shared-orderbook:** 동적 SQL 전부 placeholder 바인딩, 주문 forge/cancel `relayer==caller` 강제, settlement `submitter`는 인증주소(payload 아님), KYC 파일 path traversal 방어(`resolveWithin`+UUID), prototype pollution 방어, WebSocket broadcast-only.
- **SDK/apps:** EdDSA 키 메모리 전용·세션별 재유도·`wipeBytes`, 비밀 `getRandomValues`, `dangerouslySetInnerHTML`/`eval`/`new Function` 없음, claim 패키지 온체인 토큰 교차검증, `target=_blank` 전부 `rel=noopener`, 비밀 미로깅.

## 5. 권고

1. **A-1 우선 수정** — 공개 trade-offers를 admin과 동일 기준으로 식별자 컬럼 제거(프라이버시 invariant 일관성). 빠르고 깨끗한 fix.
2. **활성 relayer 판정에 KYC 승인 체크 포함(B-1)** — 옵션이 아닌 운영 모드 기준으로 `_isOperational`에 KYC 재검증 추가.
3. **Bond 입금 delta 기반 기록 또는 토큰 allowlist(B-2)** — 표준 ERC20만 강제.
4. **DoS 상한/게이트 적용(A-2/A-3)** — `MAX_ORDERBOOK_SIZE` 실제 적용, settlement 쓰기 registry 게이트 + 행 상한/pruning.
