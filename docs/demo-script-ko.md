# 시연 스크립트: 이더를 여러 주소로 예약 전송하기

> zkScatter의 ZK proof 기반 프라이빗 전송 기능을 단계별로 시연합니다.
> ETH를 예치한 후, 여러 개의 다른 주소에 시간 설정을 해서 전송하는 과정을 보여줍니다.

---

## 1단계: 에스크로 (예치)

**화면: Private Escrow (`/trade/private-escrow`)**

지금 보시는 것은 에스크로 화면입니다.

먼저 **로컬 폴더를 선택**합니다. zkScatter는 사용자의 비밀 정보(커밋먼트 노트, 거래 키)를 서버가 아닌 **사용자의 컴퓨터에만 저장**합니다. 이 폴더에 JSON 형식의 노트 파일이 생성되며, 이 파일을 잃어버리면 예치금을 찾을 수 없으므로 백업이 중요합니다.

토큰으로 **WETH(Wrapped ETH)**를 선택하고, 예치할 금액을 입력합니다.

"Deposit Privately" 버튼을 누르면 내부적으로 세 가지가 일어납니다:

1. ETH를 WETH로 래핑 (ERC-20 호환을 위해)
2. CommitmentPool 컨트랙트에 대한 토큰 승인(approve)
3. **Poseidon 해시 커밋먼트 생성 후 예치**

이때 온체인에 기록되는 것은 `Poseidon(ownerSecret, token, amount, salt)` 해시값 하나뿐입니다. 외부 관찰자는 이 해시만 보게 되며, **누가 얼마를 어떤 토큰으로 넣었는지 알 수 없습니다**. 모든 예치가 동일한 형태의 해시로 보이기 때문입니다.

예치가 완료되면 왼쪽 노트 목록에 새 항목이 나타납니다. "Active" 상태로 표시되며, leafIndex(머클 트리 내 위치)와 커밋먼트 해시를 확인할 수 있습니다.

### 로컬에 저장되는 노트 파일 구조

```json
{
  "note": { "ownerSecret": "...", "token": "0x...", "amount": "...", "salt": "..." },
  "commitment": "0x...",
  "tokenSymbol": "WETH",
  "amount": "1.0",
  "leafIndex": 42,
  "txHash": "0x...",
  "createdAt": 1712537600
}
```

## 2단계: 주문 생성

**화면: Private Order (`/trade/private-order`)**

다음은 주문서 생성 화면입니다.

### 거래 키 생성/해제

먼저 거래 키(Trading Key)가 필요합니다. "Generate with Wallet" 버튼을 누르면 MetaMask 서명을 통해 **EdDSA 키 쌍(Baby Jubjub 곡선)**이 생성됩니다. 이 키는 AES-GCM으로 암호화되어 로컬 폴더에 저장됩니다. 주문 서명에 사용되는 키로, 지갑 개인키와는 별개입니다.

### 커밋먼트 선택

폴더를 열면 아까 예치한 노트 목록이 표시됩니다. 이더를 전송할 것이므로 WETH 예치금을 선택합니다.

### 주문 설정

이번 시연에서는 **이더를 여러 주소로 예약 전송**하는 것이므로:

- **Sell Token**: WETH
- **Buy Token**: WETH (같은 토큰 — Scatter Direct 모드)
- **금액**: 예치한 전체 금액

같은 토큰끼리의 주문은 상대방 매칭이 필요 없이, 릴레이어가 즉시 정산합니다.

### 수수료 설정

릴레이어 수수료율을 선택합니다 (0.1%, 0.3%, 0.5%, 1% 프리셋). 이 수수료는 릴레이어가 ZK 증명 제출 및 가스비 대납의 대가로 가져가는 금액입니다. 수수료 요약에서 총 수령액(순금액)을 확인할 수 있습니다.

### 수령인 설정 (핵심 부분)

여기가 핵심입니다. **최대 10개의 수령 주소**를 추가할 수 있습니다. 각 수령인마다:

- **주소**: 받을 지갑 주소 (일반 주소 또는 스텔스 주소)
- **금액**: 해당 주소가 받을 WETH 수량
- **릴리즈 시간**: 이 시간 이후에만 인출 가능 (예: 10분 후, 1시간 후, 1일 후)

예를 들어:

| 수령인 | 금액 | 인출 가능 시간 |
|--------|------|---------------|
| 주소 A | 0.3 ETH | 10분 후 |
| 주소 B | 0.5 ETH | 1시간 후 |
| 주소 C | 0.2 ETH | 24시간 후 |

"Fill Rest" 버튼으로 남은 금액을 자동 채울 수도 있습니다.

### 릴레이어 선택 및 주문 전송

릴레이어를 선택하고 "Submit Private Order"를 누르면:

1. 각 수령인별 claim leaf 해시를 계산합니다: `Poseidon(secret, recipient, token, amount, releaseTime)`
2. 16개 leaf로 패딩하여 깊이 4의 **Claims 머클 트리**를 구성합니다
3. 주문 내용 + claims root를 포함한 해시를 **EdDSA로 서명**합니다
4. 서명된 주문을 릴레이어 API(`/api/private-orders`)로 전송합니다

이 과정에서 **블록체인 트랜잭션은 발생하지 않습니다**. 모두 오프체인입니다.

전송 완료 후 **claims 번들 JSON 파일**이 로컬에 저장되고 동시에 다운로드됩니다. 이 파일에 각 수령인의 인출에 필요한 비밀 정보가 담겨 있습니다.

## 3단계: 릴레이어의 정산 처리

릴레이어는 주문을 받으면 다음을 수행합니다:

### 검증

1. **EdDSA 서명 검증** — 주문이 위조되지 않았는지 확인
2. **수수료 및 만료 시간 검증**
3. 같은 토큰(WETH→WETH) 주문이므로 **Scatter Direct 모드**로 즉시 처리

### ZK 증명 생성

1. CommitmentPool의 모든 `CommitmentInserted` 이벤트를 조회하여 **깊이 20 머클 트리를 재구성**
2. 사용자의 예치금에 대한 **머클 경로 증명(path elements + indices)** 추출
3. **Nullifier 계산**: `Poseidon(secret, salt)` — 이중 사용 방지용
4. 수령인 claims root, 수수료, 잔액(change) 커밋먼트 등을 포함한 **회로 입력값 구성**
5. **snarkjs로 Groth16 증명 생성** (`settle.wasm` + `settle_final.zkey` 사용)

### 온체인 제출

생성된 증명을 **`PrivateSettlement.settlePrivate()`** 트랜잭션으로 온체인에 제출합니다.

온체인에서 일어나는 것:

- ZK 증명의 수학적 유효성 검증 (~200K gas)
- 커밋먼트 머클 증명 검증
- Nullifier 기록 (이중 사용 방지)
- Claims root 등록 (이후 수령인이 인출할 수 있도록)
- 수수료 차감 및 FeeVault에 적립 (릴레이어가 나중에 인출, 플랫폼 수수료 자동 차감)

**온체인 관찰자가 보는 것**: ZK proof 바이트, nullifier 해시, 새 커밋먼트 해시뿐입니다. 누가 누구에게, 얼마를, 어떤 토큰을 보냈는지는 전혀 알 수 없습니다.

## 4단계: 수령인의 인출 (Gasless Claim)

**화면: Private Claim (`/trade/private-claim`)**

마지막으로 수령인이 자금을 인출하는 화면입니다.

수령인은 아까 다운로드된 **claims 번들 JSON 파일**을 받습니다. (실제 서비스에서는 암호화된 채널로 전달)

> **보안 참고:** claim JSON이 제3자에게 유출되더라도 자금 탈취는 불가능합니다. ZK claim 회로에서 `recipient`이 public input으로 바인딩되어 있어, 누가 proof를 생성하든 토큰은 원래 지정된 수령 주소로만 전송됩니다. 다만 secret이 노출되면 해당 claim의 존재와 수령 주소를 제3자가 알 수 있으므로, 프라이버시 보호를 위해 암호화된 채널로 전달하는 것을 권장합니다.

파일을 업로드하면 번들 내 모든 claim 목록이 표시됩니다:

- Claim #1: 0.3 WETH → 주소 A (10분 후 인출 가능)
- Claim #2: 0.5 WETH → 주소 B (1시간 후 인출 가능)
- Claim #3: 0.2 WETH → 주소 C (24시간 후 인출 가능)

릴리즈 시간이 지난 claim을 선택하고 "Generate Proof & Claim" 버튼을 누르면:

### 브라우저 내 ZK 증명 생성 (~0.5초)

- 증명 내용: "나는 이 claims 트리의 특정 leaf에 대한 비밀을 알고 있다"
- 공개되는 것: claims root, nullifier, 금액, 토큰, 수령 주소, 릴리즈 시간
- 숨겨지는 것: 어떤 leaf인지 (비밀 값)

### 릴레이어를 통한 온체인 제출

1. 증명을 **릴레이어 API(`/api/private-claim`)로 전송**
2. 릴레이어가 증명을 오프체인에서 먼저 검증 (무효 증명 거부로 가스 절약)
3. nullifier 미사용 확인
4. **`PrivateSettlement.claimWithProof()`를 온체인에 제출 — 릴레이어가 가스비 대납**
5. 스마트 컨트랙트가 수령인 주소로 직접 토큰을 전송

### 왜 Gasless인가?

수령인의 새 지갑에는 ETH가 전혀 없어도 됩니다. 릴레이어가 가스비를 내고, 수수료는 정산 단계에서 이미 차감되었습니다. 새 지갑에 외부에서 ETH를 보내는 과정이 없으므로, **입금 주소와 출금 주소 사이의 연결고리가 완전히 차단**됩니다.

## 5단계: 주문 이력 확인

**화면: Private History (`/trade/private-history`)**

주문 이력 화면에서 전체 상태를 추적할 수 있습니다.

거래 키를 해제하면 릴레이어에 주문 상태를 조회합니다. 각 주문의 상태가 표시됩니다:

- **pending**: 주문 접수됨
- **matched**: 상대방과 매칭됨 (cross-token의 경우)
- **settled**: ZK 증명 제출 완료, 온체인 정산됨

주문을 클릭하면 상세 정보를 볼 수 있습니다:

- 각 claim의 인출 상태 ("Claimed" 또는 "Pending")
- claim별 트랜잭션 해시
- 미인출 claim의 JSON 파일 개별 다운로드

잔액(change)이 있다면 새로운 커밋먼트로 풀에 남아 있어 다음 거래에 사용할 수 있습니다.

## 요약

| 단계 | 동작 | 온체인에 보이는 것 |
|------|------|-------------------|
| 예치 | ETH → CommitmentPool | Poseidon 해시 1개 |
| 주문 | EdDSA 서명, 릴레이어 전송 | 없음 (오프체인) |
| 정산 | 릴레이어가 ZK proof 제출 | proof 바이트 + nullifier |
| 인출 | 수령인이 claim proof 제출 | proof 바이트 + 수령 주소 |

1 ETH를 예치해서 3명에게 시간차로 분배했지만, 온체인에서는 **예치와 인출 사이의 연결고리가 수학적으로 존재하지 않습니다**. 이것이 zkScatter의 Cryptographic Privacy입니다.
