---
title: Place an order
description: "Build an authorize half-proof and submit it through a relayer."
---

Placing an order on zkScatter means:

1. Picking an active relayer.
2. Building a Merkle proof for the deposit you want to spend.
3. Generating an `authorize` Groth16 proof binding an EdDSA-signed order
   to that leaf.
4. Submitting the proof + signature to the relayer's HTTP API.

## 1. Pick a relayer

```ts
import { loadRelayersWithApiInfo } from "@zkscatter/sdk/relayer";

const relayers = await loadRelayersWithApiInfo(
  network.contracts.relayerRegistry,
  readProvider,
);

const candidates = relayers.filter((r) => r.online && r.api);
candidates.sort((a, b) => Number(a.fee) - Number(b.fee));

const target = candidates[0];
if (!target) throw new Error("No relayer online");
```

## 2. Build the Merkle proof

Maintain an `IncrementalMerkleTree` fed by `CommitmentInserted` events:

```ts
import {
  IncrementalMerkleTree,
  COMMIT_TREE_DEPTH,
  loadCommitmentInsertedHistory,
} from "@zkscatter/sdk";

const tree = new IncrementalMerkleTree(COMMIT_TREE_DEPTH);
const history = await loadCommitmentInsertedHistory(
  readProvider,
  network.contracts.commitmentPool,
);
for (const row of history) tree.insert(row.commitment);

const merkleProof = tree.getProof(myStoredNote.leafIndex);
```

For long-lived sessions, persist the tree state and apply incremental
updates from `subscribeCommitmentInserted`.

## 3. Generate the authorize proof

```ts
import {
  generateAuthorizeProof,
  randomFieldElement,
  deriveEdDSAKey,
} from "@zkscatter/sdk/zk";

const eddsa = await deriveEdDSAKey(signer);
const nonce = randomFieldElement();

const authResult = await generateAuthorizeProof(
  {
    note: myStoredNote.note,
    leafIndex: myStoredNote.leafIndex,
    merkleProof,
    sellToken: tokenA,
    buyToken: tokenB,
    sellAmount: amountIn,
    buyAmount: amountOut,
    maxFee,
    expiry: Math.floor(Date.now() / 1000) + 600,
    nonce,
    relayer: target.address,
    eddsaPrivateKey: eddsa.privateKey,
    claims: [
      {
        secret: randomFieldElement(),
        recipient: stealthAddressForPayout,
        token: tokenB,
        amount: amountOut,
        releaseTime: 0,
      },
    ],
    newSalt: randomFieldElement(),
  },
  { wasm: "/zk/authorize.wasm", zkey: "/zk/authorize_final.zkey" },
);
```

## 4. Submit to the relayer

```ts
import { RelayerClient } from "@zkscatter/sdk/relayer";

const client = new RelayerClient(target.url);

const response = await client.submitOrder(
  {
    maker: account,
    sellToken: tokenA,
    buyToken: tokenB,
    sellAmount: amountIn,
    buyAmount: amountOut,
    maxFee,
    expiry: Math.floor(Date.now() / 1000) + 600,
    nonce: nonce.toString(16),
  },
  serializeAuthProof(authResult),
  "cover_taker",
);

if (response.status === "rejected") {
  throw new Error("Relayer rejected order");
}
```

## 5. Track status

Poll or subscribe via the relayer's history endpoint:

```ts
const orders = await client.getOrders(account);
const me = orders.find((o) => o.nonce === nonce.toString(16));
// me.status: "queued" | "matched" | "settled" | "expired" | "cancelled"
```

When `me.status === "settled"`, the matched payout is now a commitment
in the pool addressed to your stealth address. See
[Claim a payout](/guides/claim).

## Pre-flight UX

Always run `pre-sign preview` before triggering the proof so the user
sees what they're about to authorize:

```tsx
<PreSignPreview
  sellToken={tokenA}
  buyToken={tokenB}
  sellAmount={amountIn}
  buyAmount={amountOut}
  expiry={expiry}
  relayer={target.url}
  fee={target.fee}
/>
```

## Common errors

<AccordionGroup>
  <Accordion title="`Bad merkle proof`">
    Tree state diverged from on-chain. Reload `loadCommitmentInsertedHistory`
    and re-derive the proof.
  </Accordion>
  <Accordion title="`Order expired` from the relayer">
    Pick an `expiry` at least 2Ă— your worst-case prove time. On mobile,
    aim for 10 minutes minimum.
  </Accordion>
  <Accordion title="EdDSA signature rejected">
    Verify `deriveEdDSAKey` was called with the same wallet that owns
    the note. Account switches mid-flow desync the key.
  </Accordion>
